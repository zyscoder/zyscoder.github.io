Chromium Win32k system call lockdown

Background

Chromium sandbox通过利用操作系统提供的安全功能为Chromium进程提供强大的进程隔离，提供了一个纵深防御安全屏障。有了沙箱，即使处理不受信任的用户输入的Chromium进程（例如渲染器进程）遭到破坏，如果不将其与沙箱转义相结合，系统的其余部分也不会受到破坏。
渲染器进程处理大量不受信任的输入，还运行Chromium解析引擎中最复杂的部分（blink和v8），因此渲染器进程通常尽可能多地使用沙盒。
一旦攻下了渲染器进程，攻击者可以做如下几件事：

  • 他们可以完全访问渲染器进程中包含的Cookie/数据，如果共享了不同的origin（例如iFrame），则攻击者可以从其他origin读取数据（例如Cookie）。名为site isolation的项目正在努力确保，即使在受损的渲染器中运行，攻击也只能通过渲染器进程来访问绑定到该渲染器进程的origin已经可以访问的数据。
  • 攻击者的另一个途径是尝试并破browser进程，该进程被多个IPC端点暴露。例如，在pwnium 2中，Pinky Pie能够利用tcmalloc IPC消息在Linux平台上执行任意文件写入，而在pwn2own 2014中，VUPEN能够利用ClipboardHostMsg_WriteObjectsAsync触发Windows剪贴板漏洞；
  • 攻击者用来逃逸沙盒进程的另一种方法是简单地利用底层操作系统进行攻击。例如，在Linux上，futex漏洞CVE-2014-3153允许从渲染器进程升级权限。在Windows上，有许多内核漏洞被利用的案例，例如pwn2own 2013 MWR成功地利用了Win32k.sys bug逃逸chromium沙箱；

Win32k system call lockdown希望解决的是利用内核漏洞的第三种攻击途径。

在Linux里，一方面修复bug的速度相对较快；另一方面，Linux使用seccomp bpf引擎进行系统调用过滤。这允许在阻止系统调用的进程（例如渲染器进程）上放置细粒度的安全规则（如系统调用防火墙）。例如，在futex bug的情况下，可以修改seccomp bpf中chromium的规则，以便即使在有该bug的内核上，由于调用被阻止，该bug也无法被利用。

在Windows上，系统调用大致分为两种主要类型，由内核中的两个独立子系统实现。首先是nt!* 调用，由ntoskrnl.exe实现，还有win32k!* 调用，由Win32k.sys实现。Win32k调用往往与图形子系统（出于性能和历史原因，它在Windows上的内核中运行）相关联，而ntoskrnl调用更多地用于Windows NT api，例如文件访问、网络、POSIX。
在Windows上，没有细粒度的系统调用过滤，但每个系统调用都负责验证调用方的访问令牌是否允许进行调用。

Overview

创建一个新的沙盒缓解策略，该策略设置Windows 8中可用的win32k系统调用限制进程缓解策略。将此策略应用于渲染器进程。识别渲染器进程中进行的所有win32k调用，并且a）删除它们，或b）填充它们。

Detailed Design

在Windows 8中，Microsoft添加了一个新的进程缓解策略，称为PROCESS_CREATION_MITIGATION_POLICY_WIN32K_SYSTEM_CALL_DISABLE_ALWAYS_ON:

“Win32k系统调用禁用策略，如果启用，将阻止进程进行Win32k调用。”

对于不需要进行任何字体/图形工作的进程，这有效地减少内核攻击面。
在renderer进程运行的时候，除了少数几个启动renderer必要的win32k系统调用被代理以外，其他的win32k系统调用都会被拦截（详细实现详见文档）。win32k system call lock通过命令行参数–enable-win32k-renderer-lockdown公开（也就是说不加这个命令行就不会启用？）
