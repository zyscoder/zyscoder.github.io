SOK: On the Analysis of Web Browser Security

INTRODUCTION

浏览器是网络的门户，被普遍运用于银行交易、医疗记录查询等各种跟安全和隐私相关的事务。不幸的是，由于浏览器的复杂性、代码量大，很难做到bug-free。因而浏览器是攻击者非常有吸引力的一个攻击目标——2021年50%的0-day都是攻击浏览器的。

为了提高浏览器的安全性，浏览器厂商做了很多安全性相关的实践：

  1. 开源代码和浏览器架构，以及设计流程；
  2. 提供bounty，奖励0-day漏洞的提交；
  3. 主动去挖漏洞，并开发和部署云fuzzer；

然而，这些安全性设计的细节，以及对最新攻击的缓解方式，像妖术一样被写在社区里的各种博客和文章里，没有系统性的整理和总结，很难形成经验教训，为以后的发展形成指导。

本文尝试将现代web浏览器的安全性进行系统化研究。

  • 我们首先提供了四种主要web浏览器的统一模型，并使用提供的模型比较和对比它们的安全决策。
  • 其次，基于该模型，我们分析了过去10年中在每个开源浏览器中发现的安全漏洞，并展示了它们与开发新的缓解方案、漏洞赏金计划和已知的野生利用技术的关系。
  • 第三，基于我们的研究，我们传达了我们的见解和教训，以启发正在塑造web浏览器未来的研究人员和开发人员。

我们希望我们的系统化尝试能够帮助他们以整体的方式了解每个供应商的方法，从而增强他们的安全设计，以最大限度地减少安全影响和攻击面。

系统化研究过程中的challenges：

  1. 变化的target。浏览器的迭代速度快过任何其他的项目。为了总结出经验教训，本文只关注浏览器基础的设计问题和方法；
  2. 超大的size。浏览器代码量很大；此外跟浏览器相关的漏洞的资料也非常多且杂乱。本文只关注四款主流浏览器：Chrome，Firefox，Safari，Edge；并研究他们的issue列表和CVE报告；
  3. 各不相同的设计。各浏览器在处理安全问题的时候都有各自的现状和限制，本文为了对他们的安全问题做出客观的评价，忽略掉一些实现的细节，提出一个统一的模型来对他们进行比较；

本文的contribution：

  1. 根据沙箱方案比较了现代浏览器的架构；
  2. 研究了10年时间的浏览器bugs；
  3. 对这些漏洞进行了细致的分析和分类；
  4. 研究了浏览器最新的通用缓解措施；
  5. 对真实世界中的全利用链进行的研究。

MODERN BROWSER ARCHITECTURE

浏览器实行最小权限原则，将进程作为保护域。

所有的浏览器都可以分为三种进程类型：

  • browser进程：browser进程的权限和当前user的权限相同（高权限）。browser启动时向OS传递沙箱配置，用以管控所有的child进程（低权限）。browser进程是浏览器调用syscall的唯一进程。
  • renderer进程：renderer包含各种组件，是浏览器最大的bug生产者，因此renderer也被放入沙箱中。
  • task-specific进程。

此外：

  • 浏览器各进程之间通过IPC通讯，通常由browser进程进行管理；
  • 同源策略；

浏览器之间的不同之处

  • 站点隔离（Site Isolation）
    • UXSS可以不出沙箱就盗取用户数据，为了防止UXSS攻击，chrome提出Site Isolation把不同域放到不同的进程中；
    • Site Isolation也可以防一些硬件瞬时攻击；
    • Firefox有个类似的项目叫Fission；
  • JavaScript Engine
    • 各浏览器用得不一样，但是大体框架类似；
  • Rendering Engine
    • 不同浏览器有不同的实现；
    • 有一个基础的specification作为标准，但是不同浏览器的实现进度有差异；

沙箱

沙箱的权限策略分类：

  1. Privilege reduction
  2. Domain separation
  3. Attack surface reduction: 限制访问系统设备、内核或设备驱动等；

    不同操作系统的沙箱实现：

  • Windows：Windows通过使用安全令牌来限制每个进程。获得特定令牌级别的进程可以访问具有适当安全描述符级别的特权资源。例如，render进程以低完整性令牌级别运行，broker进程以中等完整性令牌级别运行，因此默认情况下，从渲染器进程到代理进程的任何写访问都将受到限制；
  • 然而，细粒度访问控制没有统一的协议。为了解决这个问题，Chrome和Firefox使用自己的IPC机制和资源相关函数上的二进制代码补丁来支持细粒度规则集。Microsoft在Windows 8中引入了AppContainer，通过添加附加到进程令牌的功能概念，对资源实施更细粒度的访问控制。Edge基于AppContainer创建了一个沙盒。从deny-by-default开始，Edge为被访问的系统资源构建了一组能力集（大概意思是什么权限的进程，可以用被调资源做到什么能力的事情吧）。Chrome还在试验基于AppContainer的沙盒。浏览器还利用各种功能来减少沙盒逃逸。例如，备用桌面和备用窗口工作站可用于缓解基于UI的攻击，如Shatter；引入了lockdown default DACL和Random Restricting SIDs，以强制实施更严格的DACL，从而使受损的沙盒进程无法访问其他沙盒进程；
  • Linux：与Windows不同，Linux沙盒主要基于seccomp、chroot和namespace。seccomp是基于eBPF语言的标准系统调用过滤器。由于默认的seccomp配置过于严格（？Overly tight），浏览器定义了自己的过滤规则。例如，Chrome将其自定义的seccomp规则应用于除broker进程之外的所有进程，每个进程的详细规则各不相同。第二，为了限制文件访问，基于Linux的浏览器沙盒利用chroot监禁。一旦一个进程被chroot限制，就无法访问文件系统的上层。例如，Firefox将chroot监禁应用于所有渲染器，只允许它们根据从broker进程获得的文件描述符访问特定文件。此外，浏览器使用namepsace为各种资源（如用户、网络和IPC）创建单独的空间。例如，创建并加入一个用户名称空间可以使沙盒进程位于单独的UID和GID中，从而有效地禁用对其他未沙盒进程的访问；
  • MacOS：虽然Windows和Linux支持各种类型的沙箱原语，但macOS支持一个特定格式的沙箱配置文件（.sb），用于描述给定进程的沙箱策略。通常，该文件提供允许访问的绝对文件路径的允许列表，并在默认情况下阻止所有其他文件访问。该配置文件还定义了访问网络和共享内存等其他资源的能力，并支持基于系统调用的过滤，比如Linux的seccomp，尽管它仅部署在Safari上；

    移动端的Sandbox

    由于移动端的内存等系统资源比较紧缺，导致他们在使用sandbox的时候显得比较微妙；比如Chrome的Site Isolation只在内存大于1.9G的设备上启用，并且只在用户需要输入密码的页面才启用；IOS上的sandbox规则也和MacOS不太一样，因为他们拥有不同的系统服务和IOKit驱动；

浏览器Exploit

浏览器攻击的目标是窃取用户的敏感数据或安装恶意软件以采取进一步行动。攻击者可以通过执行UXSS等攻击直接窃取数据，或者先执行代码，然后试图逃离沙箱。攻击者可能会试图通过攻击内核来获得系统的权限并逃离沙箱，这超出了本文的范围。多亏了各种缓解方案，攻击者需要将多个漏洞链接在一起，以获得任意执行。即使在控制流被劫持之后，由于渲染器进程在沙箱中运行，攻击者也应该在broker进程中找到另一组bug来逃离沙箱。攻击者经常试图利用系统服务、驱动程序或内核中的漏洞，而不是broker进程中的漏洞，来突破沙箱，这取决于其武库中可用的漏洞；

浏览器漏洞和缓解措施

  • Sandbox
    • Win32k lockdown
      • Renderer processes handle the greatest volume of untrusted input, and also run the most complex parts of Chromium’s parsing engine - blink and v8 - so renderer processes are typically sandboxes as much as possible.
    • Hypervisor based sandboxing

MISCs:

  • Chroot
    chroot，即 change root directory (更改 root 目录)。在 linux 系统中，系统默认的目录结构都是以 /，即以根 (root) 开始的。而在使用 chroot 之后，系统的目录结构将以指定的位置作为 / 位置;

    • 增加了系统的安全性，限制了用户的权力：
      在经过 chroot 之后，在新根下将访问不到旧系统的根目录结构和文件，这样就增强了系统的安全性。一般会在用户登录前应用 chroot，把用户的访问能力控制在一定的范围之内。
    • 建立一个与原系统隔离的系统目录结构，方便用户的开发：
      使用 chroot 后，系统读取的是新根下的目录和文件，这是一个与原系统根下文件不相关的目录结构。在这个新的环境中，可以用来测试软件的静态编译以及一些与系统不相关的独立开发。
    • 切换系统的根目录位置，引导Linux系统启动以及急救系统等：
      chroot的作用就是切换系统的根位置，而这个作用最为明显的是在系统初始引导磁盘的处理过程中使用，从初始RAM磁盘 (initrd) 切换系统的根位置并执行真正的init;

  • ACL, ACE, DACL, SACL
