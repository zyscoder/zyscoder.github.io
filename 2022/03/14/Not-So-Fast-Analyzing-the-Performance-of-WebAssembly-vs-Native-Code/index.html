Not So Fast: Analyzing the Performance of WebAssembly vs. Native Code

Abstract

现有对WebAssembly的性能测试工作的结果都表明，WebAssembly相对Native Code的Overhead大概是10%左右。
然而，这些测量工作仅仅局限于在100行上下的科学计算程序的benchmark上进行评估，并不能完全反应WebAssembly的真实使用情况。
由于在浏览器的运行环境中并不存在标准Unix API的执行环境（因为sandbox啥的），因此将更多真实的程序编译成WebAssembly并在浏览器中执行并不可行。

本文实现了一个叫BROWSIX-WASM的工具（基于emscripten，包括WebAssembly编译工具链 和 浏览器中system calls的执行环境两部分），让Unix应用程序可以不经修改编译成WebAssembly并在浏览器中执行。
本文在浏览器中对SPEC 2006 & SPEC 2017进行了性能测量，发现相比Native Code，在Chrome和Firefox中分别有55%和45%的slowdown。

Introduction

WebAssembly文档突出了几个常用示例，包括科学计算、图像编辑、视频编辑、图像识别、科学可视化、模拟、编程语言解释器、虚拟机和POSIX应用程序。
然而当前的测量工作仅限于在科学计算的计算程序上进行（PolyBenchC Benchmarks），虽然得出和native code相比只有10%以下slowdown的强劲表现，但并不能完全反应WebAssembly的真实使用情况。

SPEC CPU 2006/2017 Benchmarks适合用于WebAssembly的性能评估，因为其中8个套件属于科学计算，2个套件属于图形/视频处理，所有套件都是POSIX程序。
但是目前不大可能直接把SPEC测试套件编译成WebASsembly，因为这些测试套件需要一些系统服务，如文件系统、同步IO、进程调度等，WebAssembly和浏览器并不支持这些功能。
一种处理方式是直接修改测试套件的代码，然而这样不仅需要耗费大量时间，也会影响SPEC这种公认测试套件的有效性。

目前一种标准做法是用emscripten来编译C/C++成WebAssembly，但是emscripten只支持少数简单的系统调用，并不能扩展到大型应用程序。比如：为了支持同步IO（JavaScript只有异步IO），emscripten的MEMFS文件系统需要将整个文件系统镜像加预加载到内存中，非常占内存。

一个替代方案是BROWSIX，这是一个将未经修改的、full-featured Unix的应用程序运行到浏览器中的框架。BROWSIX实现了一个JavaScript可用的Unix kernel，对进程、文件、管道、阻塞IO以及其他Unix特性进行了全支持。
此外，它还包含了一个C++编译器（基于Emscripten）。
然而BROWSIX只支持JavaScript，并不支持WebAssembly。此外，BROWSIX还存在性能开销大的问题，这会影响对WebAssembly性能测试的结果（到底是WebAssembly自己慢，还是BROWSIX慢）。

Contribution：

  • 本文实现了一个BROWSIX-WASM，可以将Unix应用程序不经修改的编译成WebAssembly，并在浏览器环境下执行；
  • 本文开发了BROWSIX-SPEC，它扩展了BROWSIX-WASM，允许自动收集详细的定时和硬件芯片性能计数器信息，以便对应用程序性能进行详细测量；
  • 在SPEC CPU 2006和2017上评估了WebAssembly的性能。相比JavaScript，wasm速度快1.3倍；但相比native code，wasm在Chrome和Firefox中的平均slowdown为55%和45%；
  • 对WebAssembly的性能瓶颈进行了分析：
    • 与Native Code相比，WebAssembly有更大的寄存器压力：
      • 浏览器有两个保留寄存器，被用于GC等，导致WebASsembly的代码可用的寄存器数量更少；
      • 浏览器中WebASsembly编译出来的机器码，对寄存器的分配方式更烂。clang用贪心着色法进行寄存器分配，而浏览器采用线性扫描的方法进行寄存器分配；
      • 浏览器中对x86丰富的寄存器寻址方式（间接寻址、直接寻址等）并没有充分利用，导致编译出来的代码速度慢；
    • WebAssembly编译器的优化不够，导致编译出来的代码有更多无用的跳转。此外wasm本身为了做栈溢出检查、函数跳转目标检查（CFI？）等安全检查，此外这些检查导致分支跳转也更多，导致L1 cache miss率增加。
    • 为了安全检查，WebAssembly编译时额外插入了更多的代码；

*** PS：详细评估见论文 ***

Conclusion

总的来说，WebAssembly相对JavaScript确实快，但是相比Native Code并没有夸得那么好。

从瓶颈上来讲，有的原因是无法避免的：

  1. 由于浏览器保留寄存器导致可用的寄存器数量减少；
  2. 由于安全检查引入更多的代码量和更多的分支跳转；

但其他原因是可以改进的：

  1. 优化寄存器分配算法；
  2. 寄存器寻址方式优化；
  3. 代码优化，减少无用的分支跳转；
