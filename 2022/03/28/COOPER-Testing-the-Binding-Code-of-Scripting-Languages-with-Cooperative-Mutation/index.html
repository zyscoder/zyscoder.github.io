COOPER: Testing the Binding Code of Scripting Languages with Cooperative Mutation

fuzz javascript binding code
paper
发布日期:   2022-03-28

COOPER

Introduction

Binding code的作用，是为C/C++等底层的软件开发代码和JavaScript/Python等上层可供用户动态执行的脚本之间，提供参数翻译和表示转换的功能。
然而，底层软件开发和上层脚本开发是独立进行的，binding code已经被发现存在许多表示不一致、缺少安全检查等问题，带来了安全隐患。

现有工作大都采用静态分析的方法去检查某些具体的安全违例，比如隐式类型转换带来的错误。
这类工作都存在静态分析天然的不足：难以扩展以处理大规模的代码库、分析不准确带来的大量误报、缺乏可触发的PoC等。
也有人通过动态分析检测语言标准的违反。然而，它只报告由给定输入触发的错误，因此错误覆盖范围有限。(这是什么工作啊，回头瞅瞅)。

自然而然引出fuzz的概念和favocado等相关工作。
favocado只修改javascript脚本代码来触发bug，但是binding code旨在连接高级脚本语言和用低级语言编写的底层系统。因此它接收来自两个维度的输入：

  • 脚本语言代码；
  • 底层系统的输入（比如对Adobe Acrobat而言的PDF文件）；

只对其中一个维度进行变异不能发现所有的bug。例如，尽管Adobe Acrobat同时接受PDF文件和JavaScript代码作为输入，但字体等特定程序功能只能从本机输入（即PDF文件）中改变 —— 仅仅改变JavaScript代码不能触发此类错误。

解决这一限制的方法似乎很简单：我们从两个维度对输入进行变异：修改本机输入以准备程序的初始状态，以及合成有效脚本以触发更多不同的状态。然而，由于变异空间大，随机变异二维的简单方法无法有效地探索binding code：原生输入包含许多对象，但并非所有对象都与binding code相关，而脚本代码只接受特定的程序状态以进行进一步操作。

为了减少两个维度的变异空间，我们提出了cooperative mutation，它利用本地输入和脚本代码之间的关系来指导fuzz过程。具体来说，

  • 它首先修改与binding code相关的本机对象；
  • 然后，它合成脚本代码，这些脚本代码将操纵由当前本机输入触发的程序状态。

这样所有的计算资源都被用来影响binding code的执行。
然而，在采用合作突变之前，我们需要解决两个挑战。

  • 首先，推断本机输入和绑定代码之间的关系非常重要。考虑到复杂的底层系统和多样的脚本语言，手动工作对于处理真实世界的绑定代码来说是不可扩展的。
  • 其次，很难判断每个脚本代码是否接受由给定本机输入触发的程序状态（这是啥意思啊？）。这一问题在商业软件中更为严重，因为在商业软件中，源代码通常不可获得，文档也没有给出详细信息。

本文设计了三种新技术，以应对上述挑战，实现实际的合作变异。

  • 对象聚类：根据语义相似性对本地输入中的对象进行聚类。聚类有助于显著减少输入变异空间。
  • 统计关系推理：依靠大量的具体执行来统计推断本机输入和脚本代码之间的关系。具体来说，记录下每个测试文件中的对象和脚本代码，并使用应用程序打开它，以收集脚本代码的执行结果（即成功或失败）。如果执行的成功与包含特定对象高度相关，那么我们认为这些对象与脚本语句内部相关。虽然这种关系尚未得到正式确认，但大量处决提供了统计保证。评估表明，推断出的关系有助于在binding code中发现大量严重错误。
  • 关系引导变异：定义一组变异策略，使用推断出的关系来指导对象修改和脚本生成。具体来说，我们只修改与脚本代码相关的对象，并更新相应的脚本代码以生成各种运行时操作。

Motivation

Motivation Example


上图显示了一个简化的PDF文件，该文件触发Adobe Acrobat基于堆的缓冲区溢出。
PDF文件被组织为对象树。每个对象都有一个唯一的标识符，其内容在一对关键字obj和endobj中定义。
在本例中，PDF树从第2行定义的根对象1开始。
对象1包含一个Page对象2，而对象2又只有一个Kids对象3。
对象3使用对象4中定义的字体TT1。
对象4包含一个数组宽度，用于指定字符集中每个字形的宽度（第8行）。
对象3进一步指定打开/O此文件时的附加操作/AA，该文件运行对象5（第10-14行）中定义的JavaScript代码。
第12行的JavaScript语句将文件的缩放类型更改为ReflowWidth。

PDF对象和JavaScript语句之间的相互作用可能会触发此错误。具体来说，当Adobe Acrobat打开此文件时，它将按照对象3中的指令在第12行执行JavaScript语句。
JavaScript引擎调用reflow.api模块中的函数（用C++实现）。 reflow.api模块首先创建一个占用256个四字节的固定长度的堆缓冲区。
然后，它将对象4中宽度数组的内容复制到固定长度缓冲区。
但是，该文件中的数组包含257个四字节元素，因此副本将超出固定长度缓冲区的边界，并覆盖以下四个字节。
我们的手动分析显示reflow.api在内存复制之前检查宽度的长度，但由于>和>=之间的混淆，它不小心允许的边缘大小为257。 如果我们用jae（即，如果高于或等于ja，则跳转）手动替换ja指令（即，如果高于或等于ja，则跳转），程序将运行良好，不会出现任何崩溃。

为了从正常输入生成图1中触发错误的PDF文件，我们需要对PDF对象和JavaScript语句都进行修改。

  • 一方面，我们需要主动修改宽度数组以添加额外的元素。基于PDF格式参考文献，在简单字体中（例如本例中的TT1对象），每个字符代码由一个字节表示。因此，指定每个受支持字符的字形宽度的宽度数组最多可以有256个元素。作为一个示例，在从public收集的16000个PDF文件中，我们没有发现任何一个包含257个或更多元素的Widths对象。更重要的是，Adobe Acrobat没有为JavaScript代码提供修改字形宽度的接口。这意味着仅仅更新JavaScript代码永远不会触发这个错误。
  • 另一方面，将缩放类型更改为reflowW的操作会将PDF文件重新绘制为单列，其宽度与当前窗口相同（主要是为了便于阅读）。此操作不能在任何PDF本机对象中指定，我们必须借助JavaScript代码在运行时动态触发此操作。通过单击Acrobat图形界面上的相应按钮，也可以手动启用回流模式，但这将禁用高效的自动程序测试。

Challenges

虽然合作变异的想法很容易理解，但实现这个解决方案并容易。主要的挑战来自两个维度的巨大变异空间。

  • 首先，本机输入文件可以有许多对象，但并非所有对象都与绑定层相关。在图1中，关键步骤是在宽度数组中插入一个额外的元素，而现有256个元素的具体值不会影响绑定层的执行。
  • 其次，脚本代码可以组合底层系统提供的数千个API中的任意数量的函数。合成脚本应该接受底层系统通过解析本机输入创建的程序状态。否则，脚本要么无法触发新的程序状态，要么甚至无法执行。

favocado等之前的工作提出了API分组以减少代码搜索空间。具体来说，他们根据高级语义将各种API划分为几个独立的组。对于每一次变异，favocado只使用一组API合成代码。然而，API分组只是部分解决了我们的挑战，因为输入空间仍然太大，无法轻松处理。

为了减少变异空间，我们建议使用native输入和脚本代码之间的关系来指导变异。具体来说，我们修改native输入和相关脚本，以转移binding code的执行路径。输入脚本关系的指导帮助我们专注于测试binding code。然而，我们需要应对以下挑战：

  • C1：如何有效地推断本机输入和脚本代码之间的关系？
  • C2：如何利用推断出的关系从两个维度引导输入突变？

除了这两个挑战之外，我们意识到巨大的投入空间可能会首先阻碍任何有效的解决方案。我们计划遵循favocado中使用的方法，将本机输入对象也分为不同的类，这将导致第三个挑战：

  • C3：如何将本机输入对象聚类到语义相似的类？

Design

Overview

  1. COOPER解析给定的示例文档（Sample Documents）以提取本机对象。为了减少对象搜索空间（即挑战C3），COOPER根据对象的属性将对象分类为不同的类别。
    COOPER还采用了favocado中提出的方法对API进行分组。
  2. COOPER推断出object class和API group之间的关系，以应对挑战C1。具体来说，它通过组合不同的object类和API group来生成大量文档，并记录嵌入式脚本的执行结果。根据脚本执行的成功率和对象类的分布，COOPER推断出API组和对象类之间的关系。
  3. COOPER利用推断出的关系来指导对象选择、脚本生成和对象突变。我们设计了几种合作变异策略来解决挑战C2。

对象聚类

假设一个文档包含 M 个native对象，binding code支持 N 个脚本API来操作对象，理论上我们需要推断 M * N 对对象-API组合。考虑到M * N的巨大搜索空间，对每一对obj-API进行推断是不切实际的。相反，relationship实际上是在obj类和API组之间定义的。换句话说，同一类型的对象可以由同一组API操作。因此，在第一步中，我们将大量对象聚集到基于类型的类中，以简化关系推理任务。

对象属性（attributes）在主流的文档中被广泛使用，我们可以使用它来可靠地对对象进行聚类。

由于属性名提供高级语义，我们决定使用它们来对对象进行聚类。

  • 一方面，相同类型的对象在设计上应该共享公共属性名称，这是我们基于属性的聚类的理论基础。
  • 另一方面，由于可选属性（即对象可能具有或不具有该属性），即使两个对象具有相同的类型，它们也可能具有不同的属性集。在这种情况下，我们决定使用属性相似性来聚类不同的对象，而不是使用精确的属性集。

我们的聚类过程遵循两个步骤：

  1. 首先，我们为每个属性创建一个新类，并将其所有值对象放入该类中；

    对值对象进行聚类（Clustering Value Objects）：在第一步中，我们使用属性名称对其值对象进行聚类。直观地说，如果两个对象可以用作同一属性的值，那么它们必须具有相似的高级语义。具体来说，我们将检查每个（名称、对象）对：如果我们找到一个新的属性名称，我们将创建一个新类，并将对象放入该类中；如果我们看到了属性名，我们将找到现有的类并将对象放在那里。根据我们的实验，大多数obj都会被用作其他obj的value，因此该方法可以很好地处理对象。对于不用作值的对象，我们只需将它们放入一个特殊的类中。在等式2中，四个对象objectx、objecty、objectz和objectw被用作同一属性名称A的值。因此，我们将创建一个namea类，并将这四个对象放进去。

    虽然基于属性的聚类是合理的，但我们观察到两个问题会降低结果的准确性。

    1. 首先，属性名可能具有过于笼统的含义，这使得创建的类包含语义不同的对象。我们需要将这些类划分为细粒度类。例如，在图1中，attribute/AA表示附加操作，其值是包含一组具体操作的Dictionary对象。基于值对象聚类，我们将把所有这样的字典对象放在一个类中，不管它们包含/Page、/Field或其他显著不同的对象的操作。
    2. 第二，由于属性名称不同，语义相似的对象可以放入多个类中。例如，属性/Parent和/Kids在组织PDF文档时扮演类似的角色，可以由同一组脚本API进行操作。我们应该把它们放在一起。但是由于属性名称不同，它们被分为两个类。

  2. 第二，基于属性的相似性，我们将类拆分为较小的类或将它们合并为较大的类；

    属性相似性聚类：我们依靠对象之间的公共属性将现有类拆分为较小的类或将其合并为较大的类。具体来说，我们使用骰子系数来衡量相似性，如等式3所示。
    A和B是两个对象的属性集。
    直观地说，较大的骰子系数意味着A和B有更多的公共属性，这表明它们的潜在联系更强。如果sim(A，B)大于阈值θ，我们将它们放入同一类；否则，我们将把它们分为两类。可以根据经验定义阈值，以获得最佳模糊效果。在我们的实验中，我们使用两种不同的阈值进行分割和合并，以获得细粒度的聚类结果。我们将θ设置为0.3进行拆分，并使用θ=0.7进行合并。

对象类-API组关联关系

这部分很容易想到。把上面分好的对象类和API组放到一起一一进行组合，然后测试这样得到的PDF文件会不会报错，从而归纳出对象类和API组之间的关系。具体实现懒得看了= =。

Relationship-guided Mutation

总之就是像Domato一样写一下生成模板，根据前面找到的对象类-API组之间的关系，在对象的引用上和API的调用上进行变异，生成出新的Document拿去测试。
